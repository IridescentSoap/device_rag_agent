# 空管设备运维 RAG 问答技术方案（v2）

## 一、目标任务

基于设备操作手册、运维处理日志等数据构建空管设备运维知识库，通过 RAG 对用户运维问题给出**可执行、可核对**的回答。上线后与现有**自研模型**对比，验证在本领域表现更优。已知自研模型训练数据**未包含**运维处理日志，对比时需明确实验设置，避免结论失真。

**约束**：单卡推理显存 **32GB**；embedding 固定为 **bge-large-zh-v1.5**。

---

## 二、总体架构（v2）

1. **双源知识**：操作手册（规范、步骤、参数）+ 结构化运维日志（案例、处置路径）。
2. **双索引/分路召回**：手册索引与日志索引**物理或逻辑分离**，便于按查询类型加权，避免短日志淹没长手册或反之。
3. **统一检索管线**：Query 理解与路由 → BM25 + 向量双路召回 → **Qwen 系 Reranker** 精排 → 阈值/最少条数策略 → 上下文扩展 → **Qwen 7B/8B 级生成**（带引用约束）。
4. **可控生成**：回答必须绑定检索片段 ID/出处；无法支撑时输出**排查路径 + 需人工确认项**，不编造操作步骤。

---

## 三、数据处理（v2 增强）

### 3.1 操作手册（PDF/DOCX，约 1000 页）

- 使用 **MinerU**（或等价管线）解析，重点保留：**章节标题层级、页码/锚点、表格文本化、图注**。
- **递归分块**（多档对比：512 / 1024 / 2048），每块写入元数据：`doc_id`、`chapter_path`、`page_range`、`chunk_id`。
- 对**关键表格**可单独成块或附在相邻叙述块上，降低「表与文分离」导致的召回失败。
- **去重**：跨页重复页眉页脚、版权声明等做规则过滤，减少向量噪声。

### 3.2 运维日志（已结构化为 JSON，约 1.27 万条）

- 统一拼接模板（示例）：

  `[system]{system} [phenomenon]{fault_description} [impact]{fault_influence}`

  （字段名以实际 CSV 为准，避免 `falut`/`fault` 混用导致空字段。）

- 每条保留 **`case_id` / 行号** 作为引用键；若有时间、状态、处置结果标签，一并入元数据。
- **近似去重**：相似案例聚类或 SimHash，优先保留信息更全的一条，避免检索 TopK 被重复案例占满。

## 四、模型与资源选型

### 4.1 Embedding（固定）

- **bge-large-zh-v1.5**
- 要求：全库用**同一模型、同一归一化方式**建索引；手册与日志**维度一致**，便于同一向量库或双集合管理。

### 4.2 生成模型（Qwen 7B/8B）

| 选项 | 说明 |
|------|------|
| **首选：Qwen2.5-7B-Instruct** | 生态成熟、中文与指令遵循稳定，适合 RAG「按引用作答」；在 32GB 上配合量化（如 AWQ/GPTQ 或官方推荐精度）通常可单卡部署。 |
| **备选（偏「8B」档位）：Qwen3 系列 8B Instruct** | 在同等显存下略增能力边界，需以实际显存占用与延迟为准做压测；若与 Reranker、上下文长度叠加吃紧，仍以 **7B Instruct** 为默认。 |

**实践建议**：在 32GB 上优先保证「**生成长上下文 + 稳定引用**」而非盲目上大参数；**7B Instruct 作为默认基线**，8B 作为可选升级并通过同一评测集对比再定。

### 4.3 重排序（Qwen 系 Reranker）

通义 **Qwen3-Reranker** 系列（开源常见规格：**0.6B / 4B / 8B**）。

- **推荐默认：Qwen3-Reranker-4B**——精度与显存/延迟的平衡较好，适合接在双路召回之后。
- 若极致省显存或仅 CPU：**0.6B**；若离线批处理、质量优先且 GPU 有余量：**8B**。
- 云端调用时以百炼等平台文档中的 **model id**（如 `qwen3-rerank` 等）为准，与自建权重名称区分。

### 4.4 32GB 显存分配思路（原则）

- Embedding 以**离线批量**为主，不必与在线生成抢同一张卡。
- 在线链路：**Reranker（中小）+ 7B 生成** 为主；避免同卡再叠大 embedding 常驻。
- 具体是否同卡部署以压测为准，方案中保留「检索/Rerank 与生成拆分部署」选项。

## 五、知识库与检索（v2）

### 5.1 存储

- 向量库：工程实现采用 **Qdrant** 存储稠密向量（集合名：`{QDRANT_COLLECTION_PREFIX}_manual` / `{QDRANT_COLLECTION_PREFIX}_log`）；**BM25** 在本地由 `chunks.jsonl` 加载文本后构建，与 Qdrant 双路并行。需运行 Qdrant 服务并设置环境变量 `QDRANT_URL`（可选 `QDRANT_API_KEY`）。
- 说明：若需 **ES 混合检索**，可在同一套 chunk 元数据上对接，本仓库默认以 Qdrant 为准。

### 5.2 召回与融合

- **Query 拆解/轻量分类**：区分「查手册条款/参数」vs「查历史类似故障与处置」。
- **分源 TopK**：手册路与日志路各取 TopK，再合并进 Reranker，比单库全局加权更稳。
- **双路**：BM25 + embedding；融合可用 RRF 或加权后再交给 Reranker。
- **Rerank**：对合并候选统一打分，取 TopN。
- **阈值策略**：避免单一全局相似度阈值；采用「**至少保留 N 条 + 低置信提示**」或分位数/分源阈值。

### 5.3 可选增强（按需列入里程碑）

- 对难查询可试 **HyDE/子问句** 扩召回，需加「无依据不采纳」约束，防止幻觉检索。

---

## 六、上下文增强与生成（v2）

- **上下文扩展**：对命中块补充同章节相邻块或同 `chapter_path` 的上下块（控制总 token）。
- **生成 Prompt**：要求输出包含：
  - **结论/步骤**（分点、可执行）
  - **引用**（`chunk_id` / 手册页码 / `case_id`）
  - **不确定性说明**（检索不足时）
- **CoT**：可「模型内部推理」，对外输出**短而可执行**；避免冗长链式叙述。

---

## 七、评估体系（v2，支撑与自研对比）

### 7.1 指标分层

- 检索：Recall@k、nDCG（分手册/日志子集）。
- 引用：答案是否可被检索块支撑（自动规则 + 抽样人工）。
- 端到端：专家打分（可操作性、安全性、正确性）。

### 7.2 评测集

- 小规模标准 QA（如 20 条）经业务确认场景后，再扩到 200 条；**扩写需去近邻重复**，并划分 train/held-out，避免与索引语料泄漏重叠。

### 7.3 与自研模型对比（公平性）

- 至少包含：**自研裸模型** vs **RAG+本方案**；如能接入，再加 **自研+同一检索**，区分「模型能力」与「知识注入」带来的增益。

---

## 八、风险与合规（v2 必写）

- 运维答复涉及操作与安全边界：必须**引用依据**；无依据时给**通用排查框架**，不写虚构参数/步骤。
- 日志含敏感信息时：索引前脱敏规则与权限分级。

---

## 九、模型选型摘要

| 角色 | 推荐 |
|------|------|
| 生成 | 默认 **Qwen2.5-7B-Instruct**；若统一 Qwen3 且资源允许再评估 **Qwen3-8B-Instruct** |
| Rerank | 默认 **Qwen3-Reranker-4B**（显存紧用 0.6B，质量极限用 8B） |
| Embedding | **bge-large-zh-v1.5**（全库一致） |
